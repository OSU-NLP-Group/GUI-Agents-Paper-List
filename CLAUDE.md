# CLAUDE.md - GUI Agents Paper List

## Project Overview

This is a curated paper list repository for direct GUI Agent research. It tracks ~600+ papers with structured metadata across web, mobile, desktop, and general-GUI settings.

## Canonical Update Workflow

This repository is updated locally. GitHub Actions is no longer part of the official workflow.

After editing `ALL_PAPERS.md`, regenerate every derived artifact locally:

```bash
./scripts/update_repo.sh
```

This single command runs the full pipeline:
1. `normalize_institutions.py --write` — canonicalizes institution names
2. `lint_keys.py --write` — removes banned keywords, normalizes casing
3. `sort_by_date.py` — sorts papers, generates groupings and fragments
4. `assemble_readme.py` — assembles `README.md` from template + fragments

It refreshes:
- `ALL_PAPERS.md` (ordering, date normalization, institution/keyword cleanup)
- `README.md`
- `paper_by_env/`, `paper_by_key/`, `paper_by_author/`
- `update_template_or_data/` fragments and statistics

The fragment files under `update_template_or_data/` are local assembly intermediates, regenerated by the script and ignored by git.

Then review and push the generated diff normally:

```bash
git status --short
git diff --stat
git add ALL_PAPERS.md README.md paper_by_env paper_by_key paper_by_author update_template_or_data/statistics scripts tests requirements.txt .gitignore CLAUDE.md
git commit -m "..."
git push
```

Dependencies for local generation are listed in `requirements.txt`. If missing, install them with:

```bash
python3 -m pip install -r requirements.txt
```

## Repository Structure

- `ALL_PAPERS.md` — **Source of truth** for all paper entries. Edit this file to add/modify papers.
- `ADJACENT_PAPERS.md` — Adjacent but non-canonical papers that are useful to GUI-agent researchers but do not belong in the direct-GUI main list.
- `update_template_or_data/update_readme_template.md` — README template with `{{placeholders}}` filled by the local assembly step.
- `update_template_or_data/utils/scripts/sort_by_date.py` — Main script that:
  - Parses `ALL_PAPERS.md`, sorts by date (newest first), normalizes date formats, and writes back
  - Generates `recent_paper_list.md` (most recent 500 for README)
  - Generates `paper_by_env/`, `paper_by_key/`, `paper_by_author/` files
  - Generates `env_grouping.md`, `keyword_grouping.md`, `author_grouping.md`
  - Generates keyword wordcloud image
- `scripts/update_repo.sh` — Local entrypoint for regenerating all derived artifacts.
- `scripts/assemble_readme.py` — Assembles `README.md` from the template and generated fragments without mutating the template file.
- `update_template_or_data/env_grouping.md`, `keyword_grouping.md`, `author_grouping.md`, `recent_paper_list.md` — Auto-generated local assembly fragments for the README. Regenerated by the script and ignored by git.
- `paper_by_env/` — Papers grouped by environment (Web, Mobile, Desktop, General GUI). Auto-generated.
- `paper_by_key/` — Papers grouped by keyword. Auto-generated.
- `paper_by_author/` — Papers for top 20 authors. Auto-generated.
- `README.md` — Auto-generated by the local update script. Shows only the most recent 500 papers (GitHub truncates rendering at ~512KB). **Do not edit directly.**
- `requirements.txt` — Python dependencies for local regeneration.

## How to Add a Paper

Edit `ALL_PAPERS.md` and add an entry in this exact format:

```
- [title](paper link)
    - author1, author2, author3
    - 🏛️ Institutions: Institution1, Institution2
    - 📅 Date: Oct 30, 2024
    - 📑 Publisher: ICLR 2025
    - 💻 Env: [Web]
    - 🔑 Key: [framework], [dataset], [benchmark], [SomeAbbreviation]
    - 📖 TLDR: Brief summary of the paper.
```

**Important formatting rules:**
- Indentation is exactly 4 spaces for each sub-field.
- Each entry must have exactly these 7 sub-fields in this order.
- There must be a blank line between entries.
- The TLDR must end with a newline (the regex parser depends on this).

### Field Specifications

**Title:** Use the paper's canonical public title from the linked source, not an informal alias, project codename, or older draft title. If a title contains LaTeX or typography artifacts from arXiv (for example `\textsc{...}`, `$^2$`, or missing spaces after punctuation), normalize it into readable plain text while preserving the underlying wording. Keep the official abbreviation in the title when it is part of the paper title (for example `WALT`, `OS-ATLAS`, `Agent Q`), but avoid inventing abbreviations that are not present in the source title.

**Paper Link:** Prefer project homepage > arxiv link > conference link. Only one link.

**Authors:** List directly without a key identifier (e.g., `author1, author2`). Include all authors.

**Institutions:** Use common abbreviations (e.g., OSU, MIT, CMU). Use "Unknown" if not found.

**Date:** Use the earliest known public release date for this work, not the latest revision date. Prefer, in order:
1. arXiv/OpenReview v1 or initial submission/public posting date
2. first public workshop/conference page date if no earlier preprint exists
3. publisher page date only when it is the first public appearance

If the repository entry points to a later venue version but an earlier public preprint is known, the date should still reflect that earliest public release. Use canonical format `Month DD, YYYY` (e.g., `October 30, 2024`). The script will normalize other formats (like `2024-10-30`) to this canonical form. Include day if known. Year-month is acceptable (e.g., `November 2024`) — the script will default to the last day of that month for sorting but preserve the `Month YYYY` display format to indicate the exact day is unknown.

**Publisher:** Use the publication venue, not a free-form status description. Preferred format is `Venue Year` (e.g., `ICML 2024`, `NeurIPS 2025`, `ACL 2024`, `CVPR 2026`). Use `arXiv` if only on arXiv. If the presentation type matters and is publicly specified, append it in parentheses, e.g. `ICLR 2026 (Oral)`, `NeurIPS 2025 (Poster)`. Do not encode unrelated metadata into Publisher.

**Repository Scope:**
- `ALL_PAPERS.md` is for direct GUI papers only.
- Keep papers whose main contribution directly studies GUI agents, computer-use agents, GUI grounding, GUI benchmarks, GUI safety, GUI reward, GUI RL, or other GUI-native problems.
- Do not keep pure Search-API / agentic-search papers in the main list unless the paper also directly studies browser-based web agents.
- Do not keep non-GUI foundation-model papers in the main list just because later GUI work reuses them. If needed, curate those separately in `ADJACENT_PAPERS.md`.

**Adjacent Papers:**
- `ADJACENT_PAPERS.md` is optional supporting context, not part of the canonical main list.
- Use it for non-GUI-but-relevant papers such as reused foundation models, general multimodal methods, general agent infrastructure, or non-browser search-agent papers that materially inform GUI research.
- Do not mirror `ADJACENT_PAPERS.md` into `paper_by_env/`, `paper_by_key/`, `paper_by_author/`, or the README's main paper list. Those generated artifacts are sourced from `ALL_PAPERS.md` only.

**Env (Environment):**
- `[Web]` — Browser-based GUI agents interacting with real web pages
- `[Mobile]` — Android/iOS GUI agents
- `[Desktop]` — Computer-use agents on Windows/macOS/Linux
- `[General GUI]` — General GUI research not best expressed by a single concrete surface

Env should describe the paper's actual GUI interaction surface, not the repo's scope. Use concrete env tags whenever the interaction surface is known. Use multiple concrete env tags when a paper genuinely spans them (for example `[Desktop], [Mobile], [Web]`). `[General GUI]` is not shorthand for `[Desktop], [Mobile], [Web]`; use it only when the work is truly general GUI research or when the paper's contribution is not best expressed by a single concrete surface.

**Key (Keywords):**
- Each keyword wrapped in `[]`, comma-separated
- Include `[model]` if a new model is trained (not just using GPT-4 etc.)
- Include `[framework]` only when the paper proposes a novel agent framework / scaffolding / execution harness / system architecture for running agents. Do not use `[framework]` for learning frameworks, training frameworks, data frameworks, annotation frameworks, or evaluation frameworks unless the contribution is genuinely an agent runtime/scaffold.
- Include `[dataset]` when the paper releases a reusable data asset beyond a pure held-out evaluation set, such as training data, preference data, demonstrations, trajectories, or broadly reusable supervision/evaluation data. A pure benchmark with only a test/eval set does not automatically qualify for `[dataset]`.
- Include `[benchmark]` when the paper establishes a new benchmark, testbed, evaluation suite, benchmark track, or benchmark extension/augmentation that is presented as a distinct benchmark artifact. This includes both fully novel benchmarks and substantial benchmark augmentations.
- Include the paper's commonly used abbreviation (e.g., `[SeeAct]`, `[OSWorld]`)
- Include primary research focus/innovation keywords
- Avoid overly generic tags like `[web navigation]`, `[LLM agent]`, `[GUI agent]`, `[multimodal]` — these are already implied by the paper list's scope and the Env field
- Do not use env-duplicate or scope-duplicate keys such as `[web]`, `[mobile]`, `[desktop]`, `[gui]`, `[search]`, `[CUA]`, `[CUAs]`, `[web agent]`, `[web agents]`, `[mobile agent]`, `[mobile agents]`, `[desktop agent]`, `[desktop agents]`, `[gui agent]`, `[gui agents]`, `[computer-use agent]`, `[computer-use agents]`, `[computer use agent]`, `[computer use agents]`, `[computer-using agent]`, `[computer-using agents]`, `[graphical user interface]`, `[graphical user interfaces]`, `[mobile GUI agent]`, `[mobile GUI agents]`, `[browser agent]`, `[browser agents]`, `[web navigation]`

Keywords should balance retrieval value and specificity. Prefer:
- the paper's canonical abbreviation/name
- one or two artifact types (`[benchmark]`, `[dataset]`, `[framework]`, `[model]`, `[survey]`)
- one to three core technical ideas or problem settings (`[reward model]`, `[visual grounding]`, `[prompt injection]`, `[agent-as-a-judge]`)

Artifact-type guidance:
- `[benchmark]` and `[dataset]` can co-occur. Use both when a paper introduces a benchmark and also releases reusable data beyond a pure test set.
- Do not drop `[benchmark]` just because more specific technical keys are present; if the benchmark itself is a core artifact, keep `[benchmark]`.
- Be conservative with `[framework]`. In this repo it means agent framework/scaffolding, not a generic "method framework" claim from the paper text.

Avoid keywords that are just restatements of Env or scope unless they are truly central to the contribution. Do not overload Key with every benchmark or model mentioned in experiments; only include external names when they are central to understanding the paper's method or positioning.

Keyword casing should be consistent across the repo:
- General-concept keys should use lowercase words, e.g. `[visual grounding]`, `[reward model]`, `[prompt injection]`
- Official paper, benchmark, or model names should keep their canonical public casing, e.g. `[OSWorld]`, `[UI-TARS-2]`, `[SeeClick]`, `[Agent Q]`
- Do not introduce a new capitalization variant when the same key family already exists elsewhere in the repo
- Use `python3 scripts/lint_keys.py` to scan for env-duplicate keys and case-family conflicts; use `--write` to apply the repo's canonical cleanup rules

**TLDR:** 1-3 sentences, factual and contribution-first. It should answer:
1. What problem or gap is the paper targeting?
2. What does the paper actually introduce or show?
3. What is the strongest concrete outcome or takeaway?

TLDRs should be written from the paper itself, not mechanically generated. Prefer precise, bounded claims over promotional phrasing. Avoid generic openers like "This paper introduces..." when a more specific sentence is possible, and avoid overstating results beyond what the abstract/source explicitly supports.

### Searching for Paper Information

When asked to add papers, search the internet for:
- Full author list
- Exact first publication date
- Institutions (use abbreviations)
- Whether it has an arXiv version
- Whether it's accepted at a conference/journal
- Abstract and main contributions

## Workflow Details

The local update workflow:
1. Edit `ALL_PAPERS.md`
2. Run `python3 scripts/lint_keys.py --write`
3. Run `./scripts/update_repo.sh`
4. The script runs `sort_by_date.py` to sort papers and generate grouped files
5. Optionally sync arXiv-backed first-publication dates from `paper_db` with `python3 scripts/sync_dates_from_paper_db.py --write` before regeneration
6. The script assembles `README.md` from `update_readme_template.md`
7. Review the generated diff and commit/push the updated artifacts

## Development Notes

- The regex parser in `sort_by_date.py` is strict about the entry format. Malformed entries may be dropped; check the script warnings and `update_template_or_data/logs/error.log`.
- Environment field supports multi-env (e.g., `[Desktop], [Web]`) — the regex captures `\[.*?\](?:, \[.*?\])*`.
- Date strings are normalized to `Month DD, YYYY` format on each run. Month-only dates (e.g., `November 2024`) are preserved as `Month YYYY` in the output but use the last day of the month internally for sorting.
- GitHub truncates README rendering at ~512KB. The README only includes the most recent 500 papers; the full list is in `ALL_PAPERS.md`.
- Parser and generation errors are logged to `update_template_or_data/logs/error.log`. Check this file if entries seem to disappear.
- Only the top 20 keywords and top 20 authors get their own grouped files.
- `README.md` and all files under `paper_by_*/` are auto-generated — always edit `ALL_PAPERS.md` instead.
- Keyword and author matching uses exact token matching (not substring), so `[mobile]` won't match `[mobile agent]`.
- Run `python3 scripts/lint_keys.py` before regeneration if you touched keywords. Use `--write` to remove banned env-duplicate keys and normalize case-family collisions to a single repo-wide form.
